0. 운영 원칙 (전국 공통 틀)

  • 전국 공통 로직. 모든 수집·추출·가공은 전국 기준으로 설계하고, 지역은 파라미터/데이터로만 주입한다. 특정 지역 하드코딩 금지. (지역별로 소스 자체가 다른 경우만 예외 — 소스 카탈로그에 지역 필드로 명시)
  • 단일 소스 기본, 다중 소스 허용. 신호 1건의 근거는 원칙적으로 단일 공식 소스. 필요 시 복수 소스를 병합하되 sources[]에 전부 보존해 계보를 유지한다.
  • 이중 단계 관리. 모든 호재는 ① 소스상 진행단계(current_stage — 공고·고시 원문 용어)와 ② 우리 4단계 백본(phase — 계획→인가→착공→준공·개통)을 둘 다 기록한다. 매핑 규칙은 §6.
  • 수작업 최소화 → 판단의 정형화. 사람 판단은 "후보→확정 승격" 1지점으로 수렴시키되, 그 지점도 최종적으로는 거의 전부 자동화한다 — 사람 판단도 정해진 틀 안에서 이루어지므로, 판단 기준을 하나하나 명문화하면 자동 수행 가능하다(§7 로드맵). 사람의 역할은 게이트키퍼가 아니라 감사자(샘플 검수·예외 처리)로 이동한다. 신호를 손으로 새로 쓰는 일은 금지 — 수동으로 알게 된 호재는 "그 호재를 잡아낼 소스"를 카탈로그에 추가하는 방식으로 정형화한다.
  • 전국 동시 오픈 목표. (정 어려우면 수도권 우선) 강서·서부권은 검증용 예시일 뿐 오픈 단위가 아니다.

1. 파이프라인 (SSOT 3층 체인)

[수집: 자동] collect_capital.py / apt_list_all.py / build_apts_all.py └→ capital_raw.json (원시 공고·공시) + apartments_all.json (전국 단지) [후보화: 자동] signalize.py └→ signals_auto.json (후보 신호 — 5억+/DART 컷, 중복통합, 동·지번 추출, 단계 추정) [확정: 사람 1클릭] 운영자 승격 (좌표·단계 확인만) └→ signals.json (확정 신호 — 지도 정식 핀, 원시 공고 링크 보존)

사이트(index.html)는 이 3층 파일 + 경계 geojson만 읽는다. 원시→후보는 재실행으로 언제든 재현 가능해야 하며, 재현 불가능한 데이터가 확정층에 들어가면 안 된다(§2 감사 참고).

5. 가공 방법 (signalize.py — 원시→후보)

  • 컷라인: G2B는 추정가 5억+만(CUT_KRW 한 곳에서 관리), DART는 전건.
  • 중복 통합: 제목 정규화(괄호·차수·"재공고" 제거) 후 (시도, 시군구, 정규화제목 40자) 키로 그룹핑 → 같은 사업의 차수·재공고를 1신호로. 대표 공고는 최신일자.
  • 위치 추출: 제목·본문에서 동·지번 정규식 추출 → 동 대표점 폴백 좌표(법정동 경계 보유 지역). 폴백 좌표는 실좌표 아님을 항상 표기(지도 정직성 §4.1).
  • 소스단계 추정: 공고 제목 키워드 → 소스단계 라벨. 규칙표(STAGE_GUESS): 타당성|기본구상 → "타당성·기본구상 (추정)" / 기본계획 → "기본계획 (추정)" / 지구단위|정비계획 → "계획수립 (추정)" / 실시설계|설계용역 → "설계 (추정)" / 감리|건설사업관리 → "착공·공사 (추정)" / 조성|건립|신설|확장 → "사업추진 (단계 미상)"
  • 산출 필드: 제목, 기간, 공고수, 금액, 소스, 원문링크(전부), 위치추출, stage_guess.

6. 매핑 방법

6-1. 지역 매핑

regions.json(17시도·251시군구·신도시별칭·기관규칙 SSOT) + region_util.py(공용 함수). 수집범위는 active_scope.collect로만 제어 — 코드 수정 없이 전국 전환.

6-2. 분류·반경 매핑

신호의 radius_type(세부유형) 하나가 4분류(교통/일자리/정비/생활인프라)와 영향반경을 동시에 결정. 반경은 사용자 편집 가능(사이트 ⚙ 반경 설정). 점수·가중치 합산은 금지(P3) — 분류·개수·거리만.

6-3. 단계 매핑 (이중 관리: 소스단계 → 4단계 백본)

모든 신호는 소스단계(current_stage)를 원문 용어로 보존하고, 아래 규칙으로 4단계 백본(phase)에 매핑해 둘 다 기록한다. 우선순위: ① 확정층 명시 phase → ② 유형 트랙 매칭 → ③ 키워드 폴백.

7. 수동 작업 지점 · 자동화 로드맵

#수동 지점 (현재)자동화 방침
1후보→확정 승격 (좌표·단계 확인)단계적 자동화 (아래 승격 자동화 3단계) — 최종 목표는 ~100% 자동, 사람은 샘플 감사·예외만
2수동 리서치 신호 등록 (§2의 9건)중단. 새 호재를 알게 되면 신호를 쓰는 게 아니라 그 호재의 공식 소스를 sources.json에 커넥터로 추가 → 자동 유도
3지번→좌표 확인완료 — 브이월드 자동 지오코딩으로 대체
4macro 초안 관리소스 커넥터 확보 대로 폐기

승격 자동화 3단계 (판단의 정형화)

승격 판단의 실체는 5개 결정으로 분해된다: ① 진짜 개발신호인가(소음 배제) ② 위치·좌표 ③ 유형(radius_type) 분류 ④ 단계 매핑 ⑤ 기존 신호와 중복·통합. 각각을 규칙 또는 명문화된 판단 기준으로 정형화한다.

단계방식사람의 역할
1단계 (현재)사람 승격 + 자동 제안 (좌표=브이월드, 단계=STAGE_GUESS, 중복=정규화 그룹핑)건별 확인·클릭
2단계신뢰도 기반 자동 승격: 소스 공식성·위치 추출 확실성·근거 공고 수로 파이프라인 내부 품질등급 산출 → 고신뢰=자동 승격, 저신뢰=사람 큐저신뢰 큐만 처리
3단계 (목표)판단 기준의 명문화 + LLM 수행: 승격 기준(①~⑤)을 규칙·프롬프트로 완전 명문화해 자동 수행. "사람 판단도 정해진 틀 안에서 이루어진다" — 틀을 문서화하면 자동화된다주기적 샘플 감사(무작위 N건 검수) + 예외 처리만

제약조건: 자동 승격이라도 확정층 필수 요건(공식 원문 링크·좌표 근거·소스단계 원문)을 못 채우면 승격 불가 — 정직성(P2·P5)은 자동화가 흔들 수 없는 하한선. 또한 여기서 "신뢰도"는 파이프라인 내부 품질지표이며, 사용자에게 노출되는 신호·동·단지의 점수화(P3 금지)와는 무관하다.

목표 상태: 소스 커넥터(sources.json) 추가 = 데이터 1줄, 수집·후보화·승격 전자동, 사람은 샘플 감사만. 이 구조가 전국을 1인이 감당하는 유일한 방법이다.
