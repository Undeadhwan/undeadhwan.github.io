# 호재맵 데이터 표준 (v1.4 — 2026-07-17)

> **왜 필요한가.** 우리 핵심은 *호재를 정확히 발굴하고, 그 진척단계·위치를 정확히 아는 것*이다.
> 이 표준은 그 정확성을 사람의 성실성이 아니라 **절차와 자동 검사**로 보장한다.
> 모든 수집·가공 스크립트와 데이터 파일은 이 문서를 따르며, 위반은 `qa_check.py`가 배포 전에 잡는다.
>
> 제정 계기(실제 사고): 5호선 S02역과 계양강화고속도로 고촌JC가 **같은 좌표(0m)**에 찍혔다.
> 서로 다른 시설을 각각 "고촌읍 대표점"으로 지오코딩했기 때문이다. 대전 2호선 둔산역·정부청사역도 0m.
> 좌표가 "있다"는 것과 "정확하다"는 것은 다르며, 표준이 없으면 이 차이가 데이터에 조용히 쌓인다.

## 1. 3층 SSOT 체인 (불변)

```
원시(자동 수집) → 후보(자동 신호화) → 확정(승격)
```

- 확정층 승격은 아래 §5 게이트를 전부 통과해야 한다.


## 1b. 백지 재현 원칙 (v1.3 — 2026-07-17 사용자 확정)

> **"다른 웹사이트를 새로 만들어 백지에서 실행했더라도 제대로 채워져야 한다."**

`git clone` → `python3 build_all.py` → **전체 데이터 완성**. 이것이 정식 서비스의 전제다.

**핵심: 수동 입력 ≠ 재현 불가.** 조사로만 얻어지는 사실(국토부 고시 IC 명단 등)은 분명 있다.
문제는 그 사실이 **어디에 있느냐**다. 스크립트 안 딕셔너리에 있으면 재현 불가, **데이터 파일**에 있으면 재현 가능.

```
[공식 지식 — git 보관, 갱신은 이 파일을 고친다]      [자동 수집 — API/OSM]
  rail_official.json    철도 노선·역·마일스톤          arch_fetch.py    건축HUB
  official_roads.json   도로 사업·단계                 dart_fetch.py    DART 공시
  ic_names.json         IC 명단(존재=공식 근거)        seoul_hh_fetch   정비 API
  signals.json          확정 신호                      build_road.py    OSM 선형
  rail/road_osm_geom.json  OSM 선형 스냅샷
                    ↓                                          ↓
              build_all.py  →  수집·병합 → 표준 필드 → qa_check → 배포
```

**규칙**
1. **공식 사실은 데이터 파일에.** 스크립트에 사실을 하드코딩하지 않는다(코드는 로직만).
2. **외부 소스는 스냅샷을 둔다.** OSM은 접속 실패·태그 변경이 가능한 외부 의존 → `*_osm_geom.json`에 스냅샷을 보관하고 `--fetch`로만 갱신. 백지에서도 선형이 채워진다.
3. **빌더는 재실행 안전(idempotent).** 몇 번을 돌려도 같은 결과. 재실행이 수동 지식을 지우면 안 된다.
4. **갱신 = 데이터 파일 수정 + 빌더 재실행.** 공식 발표가 나오면 `official_*.json`을 고치고 `build_all.py`.
5. **검증은 백지 테스트로.** 임시 디렉터리에 스크립트+데이터만 복사해 `build_all.py` → QA 오류 0이어야 한다.

| 명령 | 용도 |
|---|---|
| `python3 build_all.py` | 공식 지식 기반 재생성 (API 없이·오프라인·수초) |
| `python3 build_all.py --fetch` | + 외부 API/OSM 재수집 (키 필요, 느림) |
| `python3 build_all.py --deploy` | + dist/dist-root 복사 (QA 통과 시에만) |

## 2. 근거 등급 `src_tier` (필수 — 모든 신호)

| 값 | 의미 | 예 |
|---|---|---|
| `official_notice` | 고시·기본계획·예타 결과 등 **공식 문서** | 국토부고시 2018-112호, 기재부 예타, 도로공사 계약정보 |
| `official_report` | 정부·지자체 **공식 발표**(및 이를 인용한 보도), 법정 공시·공공 API | 서울시 착공 보도, DART, 건축HUB |
| `osm` | **비공식** — 선형(그림) 참고용. **존재·단계·명칭의 근거가 될 수 없다** | OSM construction/proposed |
| `unverified` | 근거 미확인 | — |

**규칙**
- `osm`만을 근거로 한 신호는 **지도에 올리지 않는다**. 재조사 큐(`*_unverified_queue.json`)로 보낸다.
- OSM은 *공식 확인된 사업*의 **선형**에만 쓰고 `geom_src`로 명시한다.
- UI는 등급 배지를 반드시 노출하고, `osm`은 "단계·명칭 공식 확인 전" 경고를 함께 띄운다.


### 2b. OSM 혼입 패턴 (조사로 확인된 구조적 오탐 — 2026-07-17)

> **OSM은 "사업"이 아니라 "도로/철도 실체"를 담는다.** 이 차이가 아래 오탐을 구조적으로 만든다.
> OSM 51개 노선을 공식 대조한 결과: **실재 사업 20 / 이미 개통된 기존 도로 9 / 미확인 2**.

| 패턴 | 실제 사례 | 대응 |
|---|---|---|
| **기존 도로가 사업으로 오인** | 서낙동로·계백로·양재대로·성남이천로·봉담과천로·송산비봉로·동부대로·중봉대로·신경춘로 = **전부 이미 개통** | 개통분은 delta 아님 → `_opened_not_delta`에 기록·지도 미표시 |
| **명칭 불일치** | "익산평택고속도로"=서부내륙고속도로 / "마산거제간도로"=국도5호선 거제~마산 / "동탄-천리 지방도"=국지도84호선 중리~천리 / "신경춘로 연장"=제2경춘국도(별개 사업) | OSM 이름으로는 공식 사업을 못 찾음 → 공식 사업명이 SSOT, OSM명은 `osm_name`(매칭 키) |
| **부분 개통 노선** | 서산영덕·당진청주·호남·동해·국도38호선 = 본선 개통 + 일부만 공사중 | OSM은 `construction/proposed` 태그만 수집하므로 **건설중 구간만 선형에 남음**(다행). 단 노선 단위 핀은 금지 — 구간 단위 좌표 필수 |
| **국외 유입** | 西九州自動車道 등 일본 규슈·쓰시마 도로 5건 | 지리 규칙으로 배제(`is_foreign`) |
| **동일 사업 중복** | 광명서울=수원문산=평택파주(OSM 이름 변이 3개) | `CANON` 표로 통합 |

**결론**: OSM 단독으로는 **존재 여부조차 판정할 수 없다**. 공식 사업목록(official_roads.json)과 대조해
실재가 확인된 것만 지도에 올리고, OSM은 그 사업의 **선형(그림)**으로만 쓴다.

### 도로 접근 유형 → 영향권 규칙 (2026-07-17 사용자 확정)

> **"고속화도로(올림픽대로·내부순환로)는 IC라 안 해도 진입로를 IC처럼 호재로. 그 외 도로는 선형으로 일정범위."**

| 유형 | 판정 기준(official type) | 영향권 | 이유 |
|---|---|---|---|
| **전용** | 고속도로 · 자동차전용 · **도시고속화** | **진출입 지점(IC·진입로) 반경만** | 출입제한 — 진입로에서만 타고 내린다. 선 옆에 살아도 못 탐 |
| **일반** | 일반국도 · 지방도 · 광역도로 | **선(Line) 회랑 전체** | 연도(沿道) 접근 가능 |

**판정은 공식 유형(`official_type`)으로 한다.** OSM 이름은 부정확하다 — 실제로 국회대로·동부간선 지하화가
이름 때문에 '일반'으로 오분류돼 선 회랑을 쓰고 있었다(2026-07-17 점검에서 발견).

**진출입 지점 확보 순서**
1. **공식 IC 명단**(`ic_names.json`) — 고시·보도 근거. IC만(JC 제외).
2. 명단이 없으면 **OSM 연결로 클러스터**(`road_ramps.json`) = 진입로. **위치는 사실**(연결로가 모이는 지점),
   이름만 미상 → "…진입로(이름 미상)"로 표기. `src_tier: osm_geom`(존재·단계는 공식, 위치만 OSM 실측).
3. 둘 다 없으면 **영향권 미산정**으로 정직 표기 — 근거 없는 버퍼를 만들지 않는다.

> ⚠ 2026-07-17 점검: 전용도로 24개 중 **16개가 영향권 0**이었다(IC 명단 없음 → 호재가 지도에 있으나 집계에 안 잡힘).
> 진입로 스냅샷이 이를 메운다.

**미착공 사업의 선형**: 미착공 사업은 OSM에 `construction` 태그가 없어 **선형이 존재하지 않는다**(구조적).
사업이 공식 확인됐는데 지도에서 빠지면 안 되므로, `official_roads.json`의 `anchors`(공식 경유 대표점)로 근사 회랑을 만든다.
예: 수도권제2순환 인천~안산(송도) 구간 — 제2순환선 유일 미착공이라 OSM에 없었음(2026-07-17 사용자 지적으로 발견).

**IC 우선 규칙 (2026-07-17 사용자 확정 · 같은 날 재검토)**: 공식 IC 명단이 **존재하는** 노선은 IC 점이 그 사업의
**1차 표현**이다. 선형은 **저채도·세선 참고선으로 병행 표시**한다(`has_official_ic` → UI 감쇠 렌더).
~~선형 미표시~~는 폐기 — 수도권제2순환·세종포천 등 대형 13건이 통째로 안 보여 철도(선+역 병행 문법)와 어긋났다.
OSM 선형과 IC 점의 불일치 우려는 감쇠 표현으로 완화한다(선형은 어차피 `geom_src`에 "참고" 명시).
또한 **IC 명단의 존재 자체가 그 사업의 공식 근거**다(`ic_evidence`) — 사업목록 조사가 누락해도 IC 근거로 등재한다.

## 3c. 역(정거장) 좌표 결정 표준 (2026-07-17 제정 — 전국 공통)

**모든 역 좌표는 아래 5케이스 중 하나로 결정하고, `coord_src` 필드에 케이스를 기록한다.**
케이스가 정해지지 않은 좌표는 넣지 않는다(지도 미표시가 잘못된 표시보다 낫다 — §4.1).

| coord_src | 케이스 | 좌표 취득 | geo_prec |
|---|---|---|---|
| `gosi` | 고시·기본계획 도면에 좌표/지번 명시 | 원문 좌표 그대로 | exact/parcel |
| `existing` | 기존 역 접속·환승(예: 풍무역 환승) | **기존 역 실좌표 차용** + note "기존 ○○역 기준" | parcel |
| `shared` | 타 노선 신설역과 접속(예: 인천1 검단연장 101역) | **그 노선 데이터의 좌표 재사용**(SSOT — 같은 역이 노선마다 다른 좌표면 거짓) | 원본 따름 |
| `landmark` | 공식 문서가 지물·지구를 서술(예: "신곡IC 인근") | VWorld place(지역 키워드 검증) | landmark |
| `dong` | 위치 서술이 읍면동뿐 | 법정동 대표점 + "위치 미확정" | dong |

**금지**: 노선도 이미지 눈대중("노선도 분석 기준")으로 좌표 찍기 — 기준점 변환(georeferencing) 절차 없는
도면 추출은 좌표가 아니라 그림 해석이다. 서술 근거가 없으면 `dong`으로 내려가는 것이 정직하다.

**앵커 선형 규칙**: 역 앵커 연결(근사) 선형은 **공식 역 순서(고시·발표의 정거장 번호)**대로만 잇는다.
좌표 정밀도가 낮은 역이 섞이면 선이 실제와 달라질 수 있으므로 approx=점선 표기를 유지하고,
**급반전 QA**(연속 역 회전각 >150°·양변 >1km = 좌표·순서 오류 신호)를 통과해야 한다 —
2026-07-17 전수 스캔에서 11개 노선 14건 검출(5호선 김포검단 S05·S06 뒤엉킴 등).

## 3. 좌표 정밀도 `geo_prec` (필수 — 좌표를 가진 모든 지점)

| 값 | 뜻 | 오차 | 획득 방법 |
|---|---|---|---|
| `exact` | 원문에 좌표/지번이 명시된 실제 지점 | ~0 | 고시 도면 좌표, 기존 역 실좌표 |
| `parcel` | 지번 주소 지오코딩 | ~50m | VWorld getcoord PARCEL |
| `landmark` | 교차로·시설명 등 특정 지물 | ~200m | VWorld place 검색(지역 키워드 검증 필수) |
| `parcel_base` | 본번만 조회(부번 필지 없음) | ~260m | VWorld getcoord PARCEL — 사다리 ② |
| `station_poi` | 역·IC를 지점명으로 확인 | ~50m | VWorld place 검색 + 5km 거리 가드 — 사다리 ④ |
| `jigu` | 공식 지구경계 중심 | 지구 크기 | (훅만 존재 — 블록번호 있는 자료 확보 시 활성) |
| `zone` | 구역 폴리곤 내부 확인 | 구역 크기 | point-in-polygon 검증(정비 전용) |
| `zone_poi` | 지구명 검색 | 지구 크기 | VWorld place — **근사**, 지점 확정 아님 |
| `dong` | 읍면동 대표점 | **500m~2km** | 법정동 centroid |
| `sgg` | 시군구 대표점 | **2km+** | 시군구 centroid |
| `line` | **선형** — 점 좌표 개념 없음 | — | 노선·도로 LineString 전용 |
| `unknown` | 근거 미기록 | — | 커넥터가 출처를 안 남긴 경우 — **근사 취급** |
| `none` | 좌표 미확보 | — | 지도에 정식 핀으로 표시하지 않음 |

**규칙**
- `dong` 이하는 **"근사"로 표기**하고 지도 문법(점선·회색)을 적용한다(§4.1 지도 정직성).
- **같은 대표점을 2개 이상의 지점이 공유하면 안 된다.** 충돌 시: ① 더 정밀한 좌표를 찾는다 → ② 못 찾으면 **하나만 남기고 나머지는 "위치 미확정(지도 미표시)"** 로 둔다. 서로 다른 시설을 같은 점에 찍는 것은 거짓이다.
- **선형(LineString)에 점 정밀도를 붙이지 않는다.** 종전엔 노선 106·도로 38건이 `none`으로 찍혀
  "좌표 결손 144건"처럼 집계됐다(2026-07-19 교정) — 선형의 정직성 지표는 `geom_src`(실선형/근사 회랑)다.
- **UI 등급표(index.html `GEO_PREC`)에 없는 등급은 "확정 좌표"로 표시된다.** 등급을 추가하면 반드시
  같이 넣는다 — `unknown` 1,228건·`none` 148건이 정확한 핀처럼 보이던 P5 위반을 2026-07-19에 발견했다.
- 좌표 정밀도를 올릴 때는 **의미가 다른 지물로 대체하지 않는다**(예: "고촌읍" 대신 아무 "고촌JC"를 쓰면 안 됨 — 전자는 철도역, 후자는 도로 나들목).


## 3b. 마일스톤 `milestones[]` (필수 — 진행 단계를 가진 모든 신호)

> **왜.** 사용자가 알고 싶은 것은 "지금 착공"이 아니라 **"어디까지 왔고, 다음은 언제인가"**다.
> 기획서 P2("완료 단계=날짜, 미래=목표/미정")·§7("진척률 % 금지 → **마일스톤 트랙**")이 이미 요구하는 형태다.
> 지금까지 이 정보는 `status` 문자열에 묻혀 있었다("예타 통과('26.3 기재부) — 기본계획 착수 예정, '33 개통목표").
> 문자열은 표시만 가능하고 **정렬·비교·검증이 불가능**하다. 구조화한다.

```jsonc
"milestones": [
  {"phase":"계획", "stage":"예비타당성조사 통과", "date":"2026-03-10", "state":"done",    "src":"기재부 재정사업평가위"},
  {"phase":"인가", "stage":"기본계획 수립",       "eta":"2026-하반기",   "state":"next",    "src":"김포시"},
  {"phase":"착공", "stage":"착공",               "state":"future"},
  {"phase":"준공", "stage":"개통",               "eta":"2033",         "state":"target",  "src":"국토부 고시"}
]
```

| 필드 | 규칙 |
|---|---|
| `phase` | 4단계 백본(계획/인가/착공/준공·개통) |
| `stage` | **소스 원문 용어** 그대로("예비타당성조사 통과", "기공식", "실시계획 승인") |
| `date` | **완료된 사실만.** 소스에 명시된 날짜. 없으면 필드 자체를 두지 않는다 |
| `eta` | **소스가 명시한 목표/예정 시기만.** 우리가 추정하지 않는다(§7 정체 신호 일정 추정 금지) |
| `state` | `done`(완료·날짜 있음) / `current`(현재 단계) / `next`(다음 단계, 소스가 시기를 밝힌 경우) / `future`(미도래) / `target`(개통 목표) / `stalled`(정체·중지) |
| `src` | 그 마일스톤의 근거 기관·문서 |

**규칙**
1. **완료=날짜, 미래=목표/미정.** 미래 시점에 `date`를 쓰지 않는다. `eta`도 **소스가 말한 것만** — 없으면 "미정"으로 비워둔다.
2. **진척률(%)을 넣지 않는다.** 공정률은 `state` 판정의 내부 근거로만(도로공사 계약정보 등).
3. **다음 단계 시기를 추정하지 않는다.** "예타 통과했으니 2년 뒤 착공" 같은 계산 금지. 정체 신호에는 §7대로 **조건부 참조 소요기간**(과거 사례 통계)만 인용 가능하며, 그것도 "추정"임을 명시한다.
4. `stalled`는 P5(정직) 대상 — 숨기지 않고 표시한다(예: 울산 트램 '26.7.2 공사 중지).
5. 소스가 마일스톤을 주지 않으면 **빈 배열**로 두고 "일정 정보 없음"을 표시한다. 지어내지 않는다.

### 소스별 마일스톤 산출 규칙 (커넥터 책임)

| 소스 | 산출 방법 | 상태 |
|---|---|---|
| 건축인허가(건축HUB) | `arch_pms_day`→인가(허가) / `real_stcns_day`→착공 / `use_apr_day`→준공. **API 원문 날짜 = done** | 자동 가능 |
| DART 공시 | `rcept_dt`→계획(공시일). 투자기간 명시 시 착공·준공 eta | 자동 가능 |
| 도로(도로공사 계약정보) | 계약일자→착공(done), 공정률→`state` 판정 근거(미표시) | 자동 가능 |
| 철도(공식 조사) | 고시·보도의 날짜를 구조화(예타 통과일·기공식일·개통목표) | **조사 시 구조화 필수** |
| 정비사업 | 구역지정·조합설립·사업시행인가·관리처분·착공·준공 날짜(소스 보유) | **파서 보강 필요** |

## 4. 지점 유형 `pt_type` (필수 — 점 신호)

| 값 | 뜻 | 지도 문법 |
|---|---|---|
| `rail_station` | 철도 역·정거장 | 원(분류색 파랑) |
| `road_ic` | 도로 **IC(나들목)** — 일반 차량 진출입 가능 = **호재** | 원(분류색 파랑) — 역과 동일. 구분은 **중분류 필터**(v1.9) |
| `road_jc` | 도로 **JC(분기점)** — 고속도로끼리 연결, **진출입 불가 = 호재 아님** | **지도 미표시**(노선 상세의 구조 정보로만) |
| `facility` | 시설(사옥·병원·산단 등) | 원 |
| `zone` | 구역(정비·지구) | 원/폴리곤 |

**IC vs JC (2026-07-17 사용자 지적, 근거 명확)**: JC 옆에 살아도 그 고속도로를 탈 수 없다 — 분기점은 고속도로 간 연결일 뿐
지역 접근성을 개선하지 않는다(§7 "거리≠수혜"). 따라서 **JC는 호재가 아니다**: `signal:false` → 지도 원·영향권·집계에서 제외하고,
노선 상세에 "분기점(JC)"로 구조 사실만 표기한다. 전용도로 영향권도 **IC만** 기준으로 산정한다.
현재 공식 명단 48건 = IC 28(호재) / JC 20(제외).

**규칙**: 유형이 다른 지점은 **시각적으로 구분**되어야 한다. 교통 분류라는 이유로 역과 IC가 같은 모양이면 사용자는 둘을 구별할 수 없다.

## 5. 확정층 승격 게이트 (전부 통과해야 확정)

1. **근거**: `src_tier` ∈ {official_notice, official_report} + `sources[]`에 원문 URL
2. **좌표**: `geo_prec` 있음. `dong` 이하이면 "근사" 표기 + 충돌 없음
3. **단계**: `phase`(4단계 백본) + `current_stage`(소스 원문 용어) 둘 다 존재
4. **명칭**: 공식 명칭. 미확정이면 공식 표기 그대로(S01~S10, 101정거장) + `st_status`(확정/가칭/미정)
5. **중복**: 같은 사업이 2건으로 존재하지 않음(공간 중첩 검사 통과)
6. **delta**: 이미 개통·준공된 것(level)이 아님

## 6. 명칭 표기 규칙

- 확정 고시된 이름만 그대로 쓴다. **미확정 이름을 지어내지 않는다**(P2).
- 공식 문서가 코드만 쓰면 코드를 쓴다: `S01`, `101정거장`, `002-1정거장`.
- 가칭이면 `(가칭)`을 붙이고 `st_status: "가칭"`.
- 표시명은 **공식 사업명 우선**, 수집원(OSM 등)의 이름은 매칭용 내부값(`osm_name`).

## 7. 금지 (§7 경계와 정합)

- **진척률(%)을 표시하지 않는다.** 공정률은 단계 판정·stale 판단의 **내부 근거로만** 쓴다.
- 미확정 노선을 실선으로 그리지 않는다(점선+회랑+후보역).
- **확정 문법(전 화면 공통, 2026-07-17 일원화 · 같은 날 v1.8 재확정)**: 실선 = **착공·준공(공사라는 물리적 실체)** /
  점선 = **계획·인가(서류 단계 — 인가·고시돼도 아직 땅 위에 없다)**. 철도·도로·원·선 공통, 판정은 phase 백본으로만
  — 소스별 `confirmed` 필드는 의미가 제각각이라 지도 문법 판정에 쓰지 않는다.
  (당초 "인가부터 실선"으로 정했으나 사용자 확정으로 착공 경계로 재확정 — "인가 단계인데 실선?" 오독 사례.)
  점선 마커는 채움 투명도도 함께 낮춘다(줌아웃에서 점선이 뭉개져도 "미완성"이 읽히게).
- **분류별 표식 문법 (2026-07-17 사용자 확정 — 전면 재검토)**: **색은 4대 분류만 인코딩**한다
  (교통=파랑 / 일자리=오렌지 / 인프라=그린 / 주거개발=**마젠타 #c2255c** — 종전 레드는 오렌지와 원 크기에서 구분 불가, v1.8). 세부 유형이 색을 가지면 분류 판독이 깨진다
  — ~~도로=황갈~~은 일자리 오렌지와 혼동돼 폐기.
- **중분류 구분은 형태가 아니라 조회축 (2026-07-18 사용자 확정)**: 교통 내 철도/도로는 **중분류 필터**로 각각 조회한다.
  ~~철도=채움 / 도로=중공(흰속 도넛·흰 심선)~~ 형태 문법은 **폐기** — 대분류·중분류를 각각 필터할 수 있으면
  같은 구분을 지도 위에 이중으로 새길 이유가 없다. 지도 위 형태가 인코딩하는 것은 **단계**(실선/점선)뿐.
- 경로 미확정 순수 구상은 수록하지 않는다.
- 근거 없는 좌표를 만들지 않는다. 모르면 "위치 미확정"으로 표기한다.
