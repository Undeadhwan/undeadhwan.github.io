# 호재맵 데이터 표준 (v1.2 — 2026-07-17)

> **왜 필요한가.** 우리 핵심은 *호재를 정확히 발굴하고, 그 진척단계·위치를 정확히 아는 것*이다.
> 이 표준은 그 정확성을 사람의 성실성이 아니라 **절차와 자동 검사**로 보장한다.
> 모든 수집·가공 스크립트와 데이터 파일은 이 문서를 따르며, 위반은 `qa_check.py`가 배포 전에 잡는다.
>
> 제정 계기(실제 사고): 5호선 S02역과 계양강화고속도로 고촌JC가 **같은 좌표(0m)**에 찍혔다.
> 서로 다른 시설을 각각 "고촌읍 대표점"으로 지오코딩했기 때문이다. 대전 2호선 둔산역·정부청사역도 0m.
> 좌표가 "있다"는 것과 "정확하다"는 것은 다르며, 표준이 없으면 이 차이가 데이터에 조용히 쌓인다.

## 1. 3층 SSOT 체인 (불변)

```
원시(자동 수집) → 후보(자동 신호화) → 확정(승격)
```
- 신호를 **손으로 추가하지 않는다**. 수동으로 알게 된 호재는 그것을 잡아낼 **소스 커넥터**를 추가해 자동 유도한다.
- 확정층 승격은 아래 §5 게이트를 전부 통과해야 한다.

## 2. 근거 등급 `src_tier` (필수 — 모든 신호)

| 값 | 의미 | 예 |
|---|---|---|
| `official_notice` | 고시·기본계획·예타 결과 등 **공식 문서** | 국토부고시 2018-112호, 기재부 예타, 도로공사 계약정보 |
| `official_report` | 정부·지자체 **공식 발표**(및 이를 인용한 보도), 법정 공시·공공 API | 서울시 착공 보도, DART, 건축HUB |
| `osm` | **비공식** — 선형(그림) 참고용. **존재·단계·명칭의 근거가 될 수 없다** | OSM construction/proposed |
| `unverified` | 근거 미확인 | — |

**규칙**
- `osm`만을 근거로 한 신호는 **지도에 올리지 않는다**. 재조사 큐(`*_unverified_queue.json`)로 보낸다.
- OSM은 *공식 확인된 사업*의 **선형**에만 쓰고 `geom_src`로 명시한다.
- UI는 등급 배지를 반드시 노출하고, `osm`은 "단계·명칭 공식 확인 전" 경고를 함께 띄운다.


### 2b. OSM 혼입 패턴 (조사로 확인된 구조적 오탐 — 2026-07-17)

> **OSM은 "사업"이 아니라 "도로/철도 실체"를 담는다.** 이 차이가 아래 오탐을 구조적으로 만든다.
> OSM 51개 노선을 공식 대조한 결과: **실재 사업 20 / 이미 개통된 기존 도로 9 / 미확인 2**.

| 패턴 | 실제 사례 | 대응 |
|---|---|---|
| **기존 도로가 사업으로 오인** | 서낙동로·계백로·양재대로·성남이천로·봉담과천로·송산비봉로·동부대로·중봉대로·신경춘로 = **전부 이미 개통** | 개통분은 delta 아님 → `_opened_not_delta`에 기록·지도 미표시 |
| **명칭 불일치** | "익산평택고속도로"=서부내륙고속도로 / "마산거제간도로"=국도5호선 거제~마산 / "동탄-천리 지방도"=국지도84호선 중리~천리 / "신경춘로 연장"=제2경춘국도(별개 사업) | OSM 이름으로는 공식 사업을 못 찾음 → 공식 사업명이 SSOT, OSM명은 `osm_name`(매칭 키) |
| **부분 개통 노선** | 서산영덕·당진청주·호남·동해·국도38호선 = 본선 개통 + 일부만 공사중 | OSM은 `construction/proposed` 태그만 수집하므로 **건설중 구간만 선형에 남음**(다행). 단 노선 단위 핀은 금지 — 구간 단위 좌표 필수 |
| **국외 유입** | 西九州自動車道 등 일본 규슈·쓰시마 도로 5건 | 지리 규칙으로 배제(`is_foreign`) |
| **동일 사업 중복** | 광명서울=수원문산=평택파주(OSM 이름 변이 3개) | `CANON` 표로 통합 |

**결론**: OSM 단독으로는 **존재 여부조차 판정할 수 없다**. 공식 사업목록(official_roads.json)과 대조해
실재가 확인된 것만 지도에 올리고, OSM은 그 사업의 **선형(그림)**으로만 쓴다.

**IC 우선 규칙 (2026-07-17 사용자 확정)**: 공식 IC 명단이 **존재하는** 노선은 IC 점이 그 사업의 정확한 표현이므로
**OSM 선형을 표시하지 않는다**(`has_official_ic` → `ROAD_LINE_HIDE_WHEN_IC`). 둘을 겹쳐 그리면 서로 어긋나 보인다.
또한 **IC 명단의 존재 자체가 그 사업의 공식 근거**다(`ic_evidence`) — 사업목록 조사가 누락해도 IC 근거로 등재한다.

## 3. 좌표 정밀도 `geo_prec` (필수 — 좌표를 가진 모든 지점)

| 값 | 뜻 | 오차 | 획득 방법 |
|---|---|---|---|
| `exact` | 원문에 좌표/지번이 명시된 실제 지점 | ~0 | 고시 도면 좌표, 기존 역 실좌표 |
| `parcel` | 지번 주소 지오코딩 | ~50m | VWorld getcoord PARCEL |
| `landmark` | 교차로·시설명 등 특정 지물 | ~200m | VWorld place 검색(지역 키워드 검증 필수) |
| `dong` | 읍면동 대표점 | **500m~2km** | 법정동 centroid |
| `sgg` | 시군구 대표점 | **2km+** | 시군구 centroid |

**규칙**
- `dong` 이하는 **"근사"로 표기**하고 지도 문법(점선·회색)을 적용한다(§4.1 지도 정직성).
- **같은 대표점을 2개 이상의 지점이 공유하면 안 된다.** 충돌 시: ① 더 정밀한 좌표를 찾는다 → ② 못 찾으면 **하나만 남기고 나머지는 "위치 미확정(지도 미표시)"** 로 둔다. 서로 다른 시설을 같은 점에 찍는 것은 거짓이다.
- 좌표 정밀도를 올릴 때는 **의미가 다른 지물로 대체하지 않는다**(예: "고촌읍" 대신 아무 "고촌JC"를 쓰면 안 됨 — 전자는 철도역, 후자는 도로 나들목).


## 3b. 마일스톤 `milestones[]` (필수 — 진행 단계를 가진 모든 신호)

> **왜.** 사용자가 알고 싶은 것은 "지금 착공"이 아니라 **"어디까지 왔고, 다음은 언제인가"**다.
> 기획서 P2("완료 단계=날짜, 미래=목표/미정")·§7("진척률 % 금지 → **마일스톤 트랙**")이 이미 요구하는 형태다.
> 지금까지 이 정보는 `status` 문자열에 묻혀 있었다("예타 통과('26.3 기재부) — 기본계획 착수 예정, '33 개통목표").
> 문자열은 표시만 가능하고 **정렬·비교·검증이 불가능**하다. 구조화한다.

```jsonc
"milestones": [
  {"phase":"계획", "stage":"예비타당성조사 통과", "date":"2026-03-10", "state":"done",    "src":"기재부 재정사업평가위"},
  {"phase":"인가", "stage":"기본계획 수립",       "eta":"2026-하반기",   "state":"next",    "src":"김포시"},
  {"phase":"착공", "stage":"착공",               "state":"future"},
  {"phase":"준공", "stage":"개통",               "eta":"2033",         "state":"target",  "src":"국토부 고시"}
]
```

| 필드 | 규칙 |
|---|---|
| `phase` | 4단계 백본(계획/인가/착공/준공·개통) |
| `stage` | **소스 원문 용어** 그대로("예비타당성조사 통과", "기공식", "실시계획 승인") |
| `date` | **완료된 사실만.** 소스에 명시된 날짜. 없으면 필드 자체를 두지 않는다 |
| `eta` | **소스가 명시한 목표/예정 시기만.** 우리가 추정하지 않는다(§7 정체 신호 일정 추정 금지) |
| `state` | `done`(완료·날짜 있음) / `current`(현재 단계) / `next`(다음 단계, 소스가 시기를 밝힌 경우) / `future`(미도래) / `target`(개통 목표) / `stalled`(정체·중지) |
| `src` | 그 마일스톤의 근거 기관·문서 |

**규칙**
1. **완료=날짜, 미래=목표/미정.** 미래 시점에 `date`를 쓰지 않는다. `eta`도 **소스가 말한 것만** — 없으면 "미정"으로 비워둔다.
2. **진척률(%)을 넣지 않는다.** 공정률은 `state` 판정의 내부 근거로만(도로공사 계약정보 등).
3. **다음 단계 시기를 추정하지 않는다.** "예타 통과했으니 2년 뒤 착공" 같은 계산 금지. 정체 신호에는 §7대로 **조건부 참조 소요기간**(과거 사례 통계)만 인용 가능하며, 그것도 "추정"임을 명시한다.
4. `stalled`는 P5(정직) 대상 — 숨기지 않고 표시한다(예: 울산 트램 '26.7.2 공사 중지).
5. 소스가 마일스톤을 주지 않으면 **빈 배열**로 두고 "일정 정보 없음"을 표시한다. 지어내지 않는다.

### 소스별 마일스톤 산출 규칙 (커넥터 책임)

| 소스 | 산출 방법 | 상태 |
|---|---|---|
| 건축인허가(건축HUB) | `arch_pms_day`→인가(허가) / `real_stcns_day`→착공 / `use_apr_day`→준공. **API 원문 날짜 = done** | 자동 가능 |
| DART 공시 | `rcept_dt`→계획(공시일). 투자기간 명시 시 착공·준공 eta | 자동 가능 |
| 도로(도로공사 계약정보) | 계약일자→착공(done), 공정률→`state` 판정 근거(미표시) | 자동 가능 |
| 철도(공식 조사) | 고시·보도의 날짜를 구조화(예타 통과일·기공식일·개통목표) | **조사 시 구조화 필수** |
| 정비사업 | 구역지정·조합설립·사업시행인가·관리처분·착공·준공 날짜(소스 보유) | **파서 보강 필요** |

## 4. 지점 유형 `pt_type` (필수 — 점 신호)

| 값 | 뜻 | 지도 문법 |
|---|---|---|
| `rail_station` | 철도 역·정거장 | 원(분류색 파랑) |
| `road_ic` | 도로 IC·JC | **원 + 이중 테두리**(역과 구분) |
| `facility` | 시설(사옥·병원·산단 등) | 원 |
| `zone` | 구역(정비·지구) | 원/폴리곤 |

**규칙**: 유형이 다른 지점은 **시각적으로 구분**되어야 한다. 교통 분류라는 이유로 역과 IC가 같은 모양이면 사용자는 둘을 구별할 수 없다.

## 5. 확정층 승격 게이트 (전부 통과해야 확정)

1. **근거**: `src_tier` ∈ {official_notice, official_report} + `sources[]`에 원문 URL
2. **좌표**: `geo_prec` 있음. `dong` 이하이면 "근사" 표기 + 충돌 없음
3. **단계**: `phase`(4단계 백본) + `current_stage`(소스 원문 용어) 둘 다 존재
4. **명칭**: 공식 명칭. 미확정이면 공식 표기 그대로(S01~S10, 101정거장) + `st_status`(확정/가칭/미정)
5. **중복**: 같은 사업이 2건으로 존재하지 않음(공간 중첩 검사 통과)
6. **delta**: 이미 개통·준공된 것(level)이 아님

## 6. 명칭 표기 규칙

- 확정 고시된 이름만 그대로 쓴다. **미확정 이름을 지어내지 않는다**(P2).
- 공식 문서가 코드만 쓰면 코드를 쓴다: `S01`, `101정거장`, `002-1정거장`.
- 가칭이면 `(가칭)`을 붙이고 `st_status: "가칭"`.
- 표시명은 **공식 사업명 우선**, 수집원(OSM 등)의 이름은 매칭용 내부값(`osm_name`).

## 7. 금지 (§7 경계와 정합)

- **진척률(%)을 표시하지 않는다.** 공정률은 단계 판정·stale 판단의 **내부 근거로만** 쓴다.
- 미확정 노선을 실선으로 그리지 않는다(점선+회랑+후보역).
- 경로 미확정 순수 구상은 수록하지 않는다.
- 근거 없는 좌표를 만들지 않는다. 모르면 "위치 미확정"으로 표기한다.

## 8. 절차 (모든 데이터 변경에 적용)

```
1) 소스 확인   — sources.json에 등재된 공식 소스인가? src_tier는?
2) 수집        — 커넥터 스크립트로(수동 입력 금지). 지역은 파라미터.
3) 가공        — 표준 필드 부여: src_tier / geo_prec / pt_type / phase / st_status
4) QA 검사     — python3 qa_check.py  ← 배포 전 필수. 오류 0이어야 함
5) 배포        — dist·dist-root 동시, BUILD 갱신, cmp 정합 확인
6) 기록        — methodology.html·DATA_STANDARD.md 갱신(규칙이 바뀐 경우)
```

## 9. QA 검사 항목 (`qa_check.py`가 자동 검사)

| 검사 | 기준 | 심각도 |
|---|---|---|
| 좌표 충돌 | 유형이 다른 지점이 100m 내 | **오류** |
| 동일 노선 역 중복 | 같은 노선 내 200m 내 2역 | **오류** |
| 필수 필드 | src_tier / geo_prec / pt_type 누락 | **오류** |
| OSM 단독 근거 | src_tier=osm인 신호가 지도층에 존재 | **오류** |
| 국외 좌표 | 한국 영역 밖 | **오류** |
| 정밀도 | geo_prec=dong 이하 비율 | 경고 |
| 개통분 혼입 | phase=준공인데 delta층에 존재 | 경고 |
| 마일스톤 누락 | 진행단계 신호에 milestones[] 없음 | 경고 |
| 미래에 date | eta여야 할 미래 시점에 date 사용 | **오류** |
| 마일스톤 순서 | date가 phase 순서와 역전 | **오류** |
